## Описание
Система хранения данных (СХД) должна поддерживать механизм аутентификации при подключении инициаторов (initiators) к целевым устройствам (targets) по протоколу iSCSI с использованием:
* •	CHAP (Challenge-Handshake Authentication Protocol) — односторонняя аутентификация, при которой целевое устройство (target) аутентифицирует инициатор (initiator);
* •	Mutual CHAP (двусторонняя аутентификация) — обе стороны (initiator и target) взаимно аутентифицируют друг друга.
### 1.	Роль СХД
СХД функционирует как iSCSI Target и должна предоставлять интерфейс для настройки аутентификации на уровне:
* o	Отдельных iSCSI Target’ов;
* o	Конкретных iSCSI Initiator’ов (по их IQN).
### 2.	Поддерживаемые режимы аутентификации
СХД должна поддерживать следующие режимы:
* o	CHAP (односторонняя аутентификация):
> СХД (Target) аутентифицирует инициатор (Initiator). Initiator предоставляет имя и пароль, сконфигурированные на СХД.
* o	Mutual CHAP (двусторонняя аутентификация):
> Обе стороны аутентифицируют друг друга:
> - Initiator аутентифицируется на СХД (как в CHAP);
> - СХД (Target) также предоставляет свои учётные данные, которые Initiator должен проверить.
### 3.	Настройка CHAP на стороне СХД (Target):
Для каждого iSCSI Initiator’а (по IQN) администратор СХД должен иметь возможность:
* o	Включить/отключить CHAP-аутентификацию;
* o	Указать:
> - CHAP Username (имя пользователя инициатора);
> - CHAP Secret (секретный пароль, используемый инициатором для аутентификации);
* o	При использовании Mutual CHAP — дополнительно указать:
> - Target CHAP Username (имя СХД как Target);
> - Target CHAP Secret (секрет, который СХД предъявляет инициатору для его аутентификации).
### 4.	Учётные данные и безопасность:
* o	CHAP-секреты должны храниться в зашифрованном виде в конфигурации СХД;
* o	Шифрование должно использовать криптографически стойкие алгоритмы (например, AES-256 или эквивалент);
* o	Длина CHAP Secret должна поддерживаться от 12 до 16 символов, включая цифры, буквы и специальные символы;
* o	Передача секретов по сети запрещена — аутентификация должна выполняться по схеме challenge-response (в соответствии с [RFC1994](https://www.rfc-editor.org/rfc/rfc1994) и [RFC3720](https://www.rfc-editor.org/rfc/rfc3720)).
* o	Система должна обеспечивать защиту от атак перебора (brute-force) на этапе аутентификации (например, введение задержки после нескольких неудачных попыток).
### 5.	Привязка к инициаторам:
* o	Настройки CHAP применяются индивидуально для каждого IQN;
* o	Подключение без аутентификации должно быть запрещено, если CHAP включён для данного Initiator’а.
### 6.	Логирование:
* o	СХД должна логировать все попытки аутентификации (успешные и неудачные);
* o	В журнал должны записываться:
> - IP-адрес инициатора; (`TBD`)
> - IQN инициатора;
> - Тип аутентификации (CHAP / Mutual CHAP);
> - Результат (успех / отказ);
> - Время события.
